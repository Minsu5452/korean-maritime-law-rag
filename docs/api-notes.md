# law.go.kr OPEN API 스키마 노트

실제 API 응답으로 확인한 필드명과 파서가 다뤄야 하는 응답 변형을 기록합니다. 픽스처는 `tests/fixtures/lawsearch_sample.json`·`lawservice_sample.json`.

## 1. 요청 URL 형식

### 목록 API (lawSearch)

```
GET https://www.law.go.kr/DRF/lawSearch.do
  ?OC=${MLR_LAW_OC}
  &target=law
  &type=JSON
  &query=<법령명>   ← URL 인코딩 필요 (curl -G --data-urlencode 권장)
```

### 본문 API (lawService)

```
GET https://www.law.go.kr/DRF/lawService.do
  ?OC=${MLR_LAW_OC}
  &target=law
  &type=JSON
  &MST=<법령일련번호>
```

## 2. 목록 API 응답 필드 (lawSearch)

응답 루트 키: `LawSearch`

### `LawSearch` 메타 필드

| 필드명     | 타입   | 설명             |
|-----------|--------|-----------------|
| `law`     | array  | 법령 항목 배열    |
| `resultCode` | string | `"00"` = 성공 |
| `resultMsg`  | string | `"success"`   |
| `totalCnt`   | string | 전체 건수       |
| `page`       | string | 현재 페이지     |
| `numOfRows`  | string | 페이지 당 건수  |
| `키워드`     | string | 검색어          |
| `target`     | string | `"law"`         |
| `section`    | string | `"lawNm"`       |

### `law` 배열 항목 필드

| 필드명         | 타입   | 설명                         | 비고                          |
|--------------|--------|------------------------------|-------------------------------|
| `법령일련번호` | string | **MST** — 본문 API 핵심 키   | 코드 사용 |
| `법령ID`       | string | 6자리 영구 ID (예: `"013566"`) | 코드 사용      |
| `법령명한글`   | string | 법령명 (정식 표기)             | 코드 사용         |
| `현행연혁코드` | string | `"현행"` / `"연혁"`           | 코드 사용         |
| `시행일자`     | string | `"YYYYMMDD"` 형식             | 코드 사용         |
| `소관부처명`   | string | 부처명 (예: `"해양경찰청"`)   | 코드 사용         |
| `법령구분명`   | string | `"법률"` / `"대통령령"` 등   |                               |
| `법령약칭명`   | string | 약칭 (없으면 빈 문자열)       |                               |
| `공포번호`     | string | 공포 번호                     |                               |
| `공포일자`     | string | `"YYYYMMDD"`                 |                               |
| `제개정구분명` | string | `"타법개정"` / `"제정"` 등   |                               |
| `소관부처코드` | string | 6자리 부처 코드               |                               |
| `자법타법여부` | string | 빈 문자열이면 자법            |                               |
| `법령상세링크` | string | HTML 상세 링크 (API key 포함) |                               |
| `공동부령정보` | string | 공동 부령 정보                |                               |
| `id`          | string | 결과 내 순번 (`"1"`, `"2"` …) |                               |

> 수집·파싱 코드가 사용하는 6개 필드 모두 실제 응답에서 위 이름으로 확인했습니다.

## 3. 본문 API 응답 필드 (lawService)

응답 루트 키: `법령`

### `법령` 최상위 키

| 키        | 타입   | 설명          |
|-----------|--------|--------------|
| `기본정보` | object | 법령 헤더 정보 |
| `조문`     | object | 조문 본문     |
| `부칙`     | object/null | 부칙     |
| `개정문`   | string | 개정 이유문   |
| `제개정이유` | object | 제개정 이유  |
| `법령키`   | string | 내부 키       |

### `기본정보` 필드

| 필드명        | 타입   | 설명                   |
|-------------|--------|------------------------|
| `법령명_한글` | string | **언더스코어 포함** (lawSearch의 `법령명한글`과 다름!) |
| `법령ID`     | string | 6자리 ID               |
| `시행일자`   | string | `"YYYYMMDD"`           |
| `소관부처`   | object | `{"content": "해양경찰청", "소관부처코드": "1532000"}` |
| `법종구분`   | object | `{"content": "법률", "법종구분코드": "A0002"}` |
| `공포번호`   | string |                        |
| `공포일자`   | string |                        |
| `제개정구분` | string |                        |
| `편장절관`   | string |                        |

> **주의:** `기본정보.법령명_한글` (언더스코어)  ≠  `lawSearch.law[].법령명한글` (언더스코어 없음)

### `조문.조문단위` 배열 항목 필드

| 필드명         | 타입          | 설명                                  |
|--------------|---------------|--------------------------------------|
| `조문여부`    | string        | `"조문"` (본문) 또는 `"전문"` (장·절 헤더) |
| `조문번호`    | number/string | 조 번호 (정수)                         |
| `조문가지번호` | string        | 조의2 등의 `"2"`, 없으면 키 자체가 없음  |
| `조문제목`    | string        | 조 제목 (없으면 키 자체가 없음)           |
| `조문내용`    | string        | 조문 전체 텍스트 (조 제목 포함)          |
| `항`          | array or dict | **아래 응답 변형 참조**                    |
| `조문시행일자` | string        | `"YYYYMMDD"`                         |
| `조문변경여부` | string        | `"Y"` / `"N"`                        |
| `조문참고자료` | string/array  | `[본조신설 YYYY.M.D]` 형태            |
| `조문키`      | string        | 7자리 내부 키                          |
| `조문이동이전` | string        | 이동 전 위치                           |
| `조문이동이후` | string        | 이동 후 위치                           |

### `항` 배열 항목 필드

| 필드명   | 타입          | 설명                  |
|---------|---------------|----------------------|
| `항번호` | string        | `"①"`, `"②"` …       |
| `항내용` | string        | 항 본문               |
| `호`    | array or dict | **아래 응답 변형 참조**    |

### `호` 배열 항목 필드

| 필드명   | 타입          | 설명                  |
|---------|---------------|----------------------|
| `호번호` | string        | `"1."`, `"2."` …     |
| `호내용` | string        | 호 본문               |
| `목`    | array or dict | 있을 경우 하위 목 구조  |

### `목` 배열 항목 필드

| 필드명   | 타입   | 설명                  |
|---------|--------|----------------------|
| `목번호` | string | `"가."`, `"나."` …    |
| `목내용` | string | 목 본문               |

## 4. 응답 변형 목록

### Q1: `항`/`호`/`목` 단일 요소 객체화 (중요)

XML→JSON 변환 시 배열 원소가 1개이면 **배열이 아닌 단일 dict**으로 내려옵니다.

- `항`이 1개 → `조문단위.항` 가 `dict` (수산업법 조문단위[2] 등에서 확인됨)
- `호`가 1개 → `항.호` 가 `dict` (`항`과 동일하게 처리)
- `목`이 1개 → `호.목` 가 `dict` (`항`과 동일하게 처리)

**파서에서 반드시 `isinstance(val, dict)` 체크 후 `[val]`로 감싸야 합니다.**

### Q2: `기본정보.법령명_한글` vs `lawSearch.법령명한글` 필드명 불일치

목록 API: `법령명한글` (언더스코어 없음)  
본문 API: `법령명_한글` (언더스코어 있음)

### Q3: `조문가지번호` 키 부재

조의2 같은 경우 `조문가지번호` 필드가 있고 값이 `"2"`입니다. 일반 조의 경우 **키 자체가 없습니다**(빈 문자열이 아님). 파서에서 `.get("조문가지번호", "")`로 처리해야 합니다.

### Q4: `조문제목` 키 부재

`조문여부: "전문"` (장·절 헤더) 항목에는 `조문제목` 키가 없습니다. `.get()`으로 처리해야 합니다.

### Q5: `소관부처`, `법종구분` 중첩 객체

`기본정보.소관부처` = `{"content": "해양경찰청", "소관부처코드": "..."}` — 단순 문자열이 아닌 중첩 dict입니다.

### Q6: `목` 구조는 해양경찰법 픽스처에 없음

해양경찰법(소규모 법령)에는 `목` 레벨이 없습니다. **수산업법** (MST=270747) 실호출로 `목` 구조를 확인했습니다. 픽스처는 해양경찰법 기준입니다.

## 5. 현재 사용 범위

현재 코퍼스의 법령 목록은 `configs/laws.yaml`에 있으며, 법령별 조문 수와 인용 그래프 규모는 `reports/corpus_profile.md`에서 확인합니다. 이 문서는 API 응답 구조와 파서 구현상 주의점만 기록합니다.
